Design Patterns

Behavioral Pattern

Strategy Pattern

HTTP method handlers are interchangeable strategies behind a common interface. Adding a new HTTP method means writing one class - zero changes to the request routing core.

Open/Closed Principle - extend without modifying core logic
Class Hierarchy
class IMethodHandler {
public:
    virtual HttpResponse handle(
        const HttpRequest& req,
        const ServerConfig& cfg
    ) = 0;
};

// Concrete strategies
class GetHandler    : public IMethodHandler;
class PostHandler   : public IMethodHandler;
class PutHandler    : public IMethodHandler;
class DeleteHandler : public IMethodHandler;
class HeadHandler   : public IMethodHandler;
Behavioral Pattern

State Pattern

HTTP parsing transitions through discrete states. Each state handles its own logic and decides when to transition - making partial reads and chunked data natural.

Handles partial TCP reads gracefully - no buffering hacks
Parse State Machine
class IParseState {
public:
    virtual ParseResult parse(
        const std::string& buffer,
        HttpRequest& req
    ) = 0;
};

// State transitions:
RequestLineState  ->  GET /path HTTP/1.1
  |

  |
BodyState         ->  Content-Length body
  |
ChunkedState     ->  Chunked encoding
Creational Pattern

Factory Pattern

All HTTP responses are built through ResponseFactory - ensuring consistent headers, status lines, and content formatting across the entire server.

Single source of truth for response construction
Response Factory
class ResponseFactory {
public:
    static HttpResponse createSuccess(
        int code, const std::string& body
    );
    static HttpResponse createRedirect(
        int code, const std::string& url
    );
    static HttpResponse createError(
        int code, const ServerConfig& cfg
    );
    static HttpResponse createCGI(
        const std::string& output
    );
};
Behavioral Pattern

Observer Pattern

The ConnectionManager acts as a subject, notifying registered observers when I/O events fire. This decouples event detection from handling completely.

Decoupled event callbacks - handlers know nothing about epoll
Event Notification
class ConnectionManager {
private:
    std::map<int, IHandler*> handlers;

public:
    void registerHandler(int fd, IHandler* h);
    void unregisterHandler(int fd);

    // Notifications dispatched by event type
    void notifyReadReady(int fd);
    void notifyWriteReady(int fd);
    void notifyCGIComplete(int fd);
};
Architectural Pattern

Reactor Pattern

The master pattern binding it all together. The Poller (epoll wrapper) centralizes all I/O detection, while the EventLoop dispatches to appropriate handlers - entirely non-blocking.

Single-threaded yet handles thousands of concurrent connections
Reactor Flow
// The three pillars of our reactor:

Poller
  |  wraps epoll, centralizes I/O detection
  v
EventLoop
  |  dispatches events to handlers
  v
Handlers
     process data, never block

// Why epoll over select/poll?
select()  O(n)  scan all fds every call
poll()    O(n)  same, no fd limit
epoll()   O(1)  kernel maintains ready list
